Homelab Updates: New Proxmox Cluster

April 15, 2025

Finally got my expanded homelab cluster working with Proxmox! After months of collecting hardware, I've upgraded from my previous shoe-shelf setup to a proper 12U rack. The power bill might kill me, but it's worth it for the learning experience.

Current setup:

  • 3× Dell R720 (32GB RAM each, 2× E5-2650v2)
  • Synology DS920+ (16GB RAM, 4× 8TB WD Reds in SHR)
  • FritzBox 7590 as main router
  • Raspberry Pi cluster (3× Pi 4 8GB) for lightweight services

Unlike my previous network setup where I was using Pihole on a single Pi, I've now containerized most services and deployed them across the cluster for redundancy. The biggest challenge was setting up proper networking across all nodes:

Experimenting with Phoenix LiveView

April 2, 2025

Been diving into Elixir's Phoenix framework lately, specifically LiveView. After years of React development, the simplicity of building reactive applications without tons of JavaScript is refreshing.

Here's a simple counter component I built to test performance:

defmodule DemoWeb.CounterLive do
  use Phoenix.LiveView

  def mount(_params, _session, socket) do
    {:ok, assign(socket, count: 0)}
  end

  def render(assigns) do
    ~H"""
    <div>
      <h1>Count: <%= @count %></h1>
      <button phx-click="increment">+</button>
      <button phx-click="decrement">-</button>
      <button phx-click="reset">Reset</button>
    </div>
    """
  end

  def handle_event("increment", _, socket) do
    {:noreply, update(socket, :count, &(&1 + 1))}
  end

  def handle_event("decrement", _, socket) do
    {:noreply, update(socket, :count, &(&1 - 1))}
  end

  def handle_event("reset", _, socket) do
    {:noreply, assign(socket, count: 0)}
  end
end

Planning to use this for a monitoring dashboard for my homelab. Phoenix's lightweight nature means I can run it on one of my Raspberry Pis without issue.

Updated PGP Key Setup with Subkeys

March 10, 2025

Finally got around to reorganizing my PGP setup with proper subkeys and separation of signing, encryption, and authentication functions. This is much more secure than my previous single-key approach.

I'm now using a designated signing subkey for my open source contributions and a separate encryption subkey for secure communications. The master key remains offline on an air-gapped system.

For travel security (especially when heading to the US), I'm using a temporary travel key set that's linked to my main identity but can be revoked without compromising my primary keys.
